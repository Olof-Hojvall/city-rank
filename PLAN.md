# City Rank — Implementation Plan

A pure-frontend single-page app for ranking cities (A–F) on a fast map. State lives entirely in the URL — no backend, no accounts.

---

## 1. Goals & constraints

- **Pure frontend.** Static site, deployable to GitHub Pages / Netlify / Cloudflare Pages.
- **Fast map** that can render thousands of city markers smoothly.
- **Top-N by population** filtered to the current viewport, plus all of the user's ranked cities (always shown regardless of zoom).
- **Ranks A–F** per city. Ranking is the only persistent state.
- **State in URL** — copy URL = save. No localStorage primary store (use it only as a soft autosave).
- **Stable IDs** so the same city across sessions / shared URLs maps to the same record.
- **Export** ranked list as Markdown / CSV / plain text.

---

## 2. Tech stack

| Concern | Choice | Why |
|---|---|---|
| Build tool | **Vite** + React + TypeScript (via `shadcn` CLI) | Scaffolded by `shadcn init --template vite`; static output |
| Package manager | **pnpm** | Faster, content-addressed store; matches the `pnpm dlx shadcn` flow |
| UI components | **shadcn/ui** (preset `b0`) | Copy-in components, owned source, Tailwind-native, good defaults for buttons/dialogs/sheets |
| Styling | **Tailwind CSS** (set up by shadcn) | Comes with the shadcn init |
| Map | **MapLibre GL JS** | WebGL, free, no API key required for OSM raster; handles 10k+ points via GeoJSON + clustering |
| Tiles | **OSM raster** (dev) → **Protomaps** or **MapTiler free tier** (prod) | Free, no token needed for OSM; Protomaps `.pmtiles` is fully self-hostable |
| State | **Zustand** | Tiny, no boilerplate, easy to subscribe selectively |
| URL compression | **lz-string** (`compressToEncodedURIComponent`) | URL-safe, ~70% smaller than raw JSON |
| Cities dataset | **GeoNames `cities5000`** (~55k cities ≥ 5,000 pop) | Free (CC BY 4.0), stable `geonameid`, includes lat/lng/population/country |

> If 55k feels too big for the bundle, switch to `cities15000` (~26k). If we want to support village-level ranking, switch to `cities1000` (~150k) and load as a `.pmtiles` or chunked binary instead of JSON.

## 3. Data model

### City (static, bundled)
```ts
type City = {
  id: number;        // GeoNames geonameid — canonical, dedup key
  name: string;
  country: string;   // ISO-2
  admin1?: string;   // state/region code
  lat: number;
  lng: number;
  pop: number;
};
```

Stored as a single `cities.json` (or a packed `Float32Array` + string table for size), generated by a build script from the GeoNames TSV dump.

### Ranking (user state, in URL)
```ts
type Grade = 'A' | 'B' | 'C' | 'D' | 'E' | 'F';
type Rankings = Record<number /* geonameid */, Grade>;
```

URL shape: `https://.../#r=<lz-string-compressed-json>`
- Use the **hash** (not query string) so the server never sees it and SPA routing stays simple.
- Versioned envelope: `{ v: 1, r: { "2643743": "A", ... } }` — lets us migrate the schema later.

## 4. Performance approach for many markers

The naive "DOM marker per city" approach dies around ~1k markers. Plan:

1. **GeoJSON source + circle layer** in MapLibre. WebGL renders 50k circles smoothly.
2. **Two layers**:
   - `cities-top` — viewport-filtered top N by population (e.g. N = 200 at low zoom, 1000 at high zoom). Recomputed on `moveend`.
   - `cities-ranked` — every ranked city, always visible, colored by grade.
3. **No DOM markers.** Click handling via MapLibre's `queryRenderedFeatures` on click.
4. **Circle radius** interpolated from `pop` and current zoom: bigger cities = bigger dots at low zoom; small cities only appear once you zoom in.
5. Optional later: switch to **deck.gl ScatterplotLayer** if we ever need 100k+.

Algorithm for top-N filter:
- Pre-sort the cities array by `pop` descending (once, at load).
- On `moveend`: walk the sorted array, keep first N whose `[lat,lng]` falls in the current bounds, push as GeoJSON. O(total) but on a typed-array-backed list this is sub-millisecond for 55k.

## 5. URL state

```ts
// utils/urlState.ts
import LZString from 'lz-string';

export function encodeRankings(r: Rankings): string {
  return LZString.compressToEncodedURIComponent(JSON.stringify({ v: 1, r }));
}
export function decodeRankings(s: string): Rankings {
  const raw = LZString.decompressFromEncodedURIComponent(s);
  if (!raw) return {};
  const parsed = JSON.parse(raw);
  return parsed.v === 1 ? parsed.r : {};
}
```

- On every rank change → debounced (250 ms) `history.replaceState` to update `#r=...` without flooding history.
- On load → parse `location.hash` into store.
- Soft backup: also write to `localStorage['city-rank:last']` so an accidental tab close without copying the URL is recoverable.

**Size budget:** ~16 chars per ranked city after compression. 500 ranked cities ≈ 8 KB encoded — within all browser URL limits. If users rank thousands, we'll add a "this URL is getting long, consider exporting" warning.

## 6. UI layout

Two-column layout. Map fills the left ~60–65%, the right column is a scrollable sidebar with two stacked sections.

```
┌──────────────────────────────────┬───────────────────────────────┐
│ ◰ City Rank                      │ Suggested (current view)      │
│                                  │ ─────────────────────────────  │
│                                  │  Tokyo          37.4M    [▾]  │
│                                  │  Delhi          32.9M    [▾]  │
│                                  │  Shanghai       28.5M    [▾]  │
│             MAP                  │  São Paulo      22.4M    [▾]  │
│                                  │  ...                           │
│        (left ~60–65%)            │                                │
│                                  │ Your rankings (12)            │
│                                  │ ─────────────────────────────  │
│                                  │ ▸ A — Stockholm, Tokyo        │
│                                  │ ▸ B — Berlin, Lisbon          │
│                                  │ ▸ C — ...                      │
│                                  │                                │
│                                  │ [Export] [Copy URL] [Reset]   │
└──────────────────────────────────┴───────────────────────────────┘
```

**Right column, top — "Suggested cities":**
- Shows the same top-N-in-viewport list the map's `cities-top` layer is rendering. Sorted by population descending.
- Updates on `moveend` (debounced) so panning/zooming re-ranks the list.
- Each row: name · country · population · inline rank picker (A–F dropdown or popover with `Button` group from shadcn).
- Hover row → highlight that city's circle on the map. Click row → fly-to + select.
- Already-ranked cities in the suggestion list show their current grade as the active state of the picker.
- v2 ideas (out of scope for v1): toggle sort by name / by distance from viewport center; filter by min-population slider.

**Right column, bottom — "Your rankings":**
- Grouped by grade A→F with counts.
- Each city row: name · country · grade chip · `×` to unrank.
- Click row → fly-to + select on map.
- Always visible regardless of map viewport (this is the persistent state).

**Selected city:**
- When a map circle is clicked, the corresponding row in either list scrolls into view and gets a highlighted state. No separate "details panel" — the row itself is the detail.

**Components from shadcn to pull in:**
`button`, `card`, `scroll-area`, `separator`, `dialog`, `dropdown-menu` (or `popover` for the rank picker), `toggle-group` (clean A–F picker), `sonner` (toasts for "URL copied"), `tooltip`.

**Color palette per grade:** A = green, B = lime, C = yellow, D = orange, E = red-orange, F = red. Drive both the map circle color and the grade chips from the same constant in `src/lib/grades.ts` so they stay in sync.

- **Markdown** — `## A\n- Stockholm, SE\n- Tokyo, JP\n\n## B\n...`
- **CSV** — `grade,name,country,population,lat,lng`
- **Plain text** — same as markdown without the `##`/`-`

Open in a modal with copy-to-clipboard + download-as-file.

## 10. Open questions / decisions to revisit

- **Dataset size.** Start with `cities5000`. Re-evaluate after first perf test.
- **Tile provider for prod.** OSM raster is fine to start but has usage limits; Protomaps `.pmtiles` self-hosted is the cleanest long-term answer.
- **Mobile UX.** Side panel becomes a bottom sheet. Defer to v2.
- **Sharing UX.** "Copy URL" is the v1 share. A `?shared=1` flag could enable a read-only mode later.
- **Diff/merge.** If two people rank separately and want to merge, that's a v2 feature — likely a "load second URL and pick conflicts" dialog.

---

## 11. Out of scope (v1)

- Accounts, sync, multi-device.
- Comments / notes per city.
- Custom rank schemes (always A–F).
- Heatmaps, choropleths, country aggregates.
- i18n.
